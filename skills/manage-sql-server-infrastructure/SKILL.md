# Manage SQL Server Infrastructure

Create or refactor a **DB-first** SQL Server infrastructure project aligned with this repository's DDD .NET conventions. Start from SQL, keep **EF Core Power Tools** as the source of truth for scaffolded EF artifacts, and keep Domain contracts typed on business/domain models instead of scaffolded EF entities.

## When to Use This Skill

- Add a new `Client.Project.SqlServer` infrastructure project
- Align an existing SQL Server infrastructure project to this repository style
- Refactor Domain contracts so they depend on domain/business models, not EF scaffolded entities
- Standardize batch read/write behavior, retries, configuration, and DI registration

## Target Structure

```text
src/
├── Domains/
│   └── Client.Project.Domain/
│       ├── Common/
│       │   ├── Interfaces/
│       │   │   └── IRepository.cs
│       │   └── Results/
│       │       ├── RepositoryBatchResult.cs
│       │       └── RepositoryPagedBatchResult.cs
│       ├── Models/
│       │   ├── Customer.cs
│       │   └── Order.cs
│       └── Repositories/
│           ├── ICustomerRepository.cs
│           └── IOrderRepository.cs
└── Infrastructures/
    └── Client.Project.SqlServer/
        ├── Abstractions/
        │   └── SqlServerBaseRepository.cs
        ├── Configurations/
        │   └── SqlServerConfig.cs
        ├── Context/
        │   └── <Generated by EF Core Power Tools>
        ├── Entities/
        │   └── <Generated by EF Core Power Tools>
        ├── Mappings/
        │   └── <Entity>Mappings.cs
        ├── Repositories/
        │   ├── CustomerRepository.cs
        │   └── OrderRepository.cs
        └── ServiceCollectionExtensions.cs
```

## Naming Guidance

- Infrastructure project name: `Client.Project.SqlServer`
- Configuration class: `SqlServerConfig`
- Shared base class: `SqlServerBaseRepository`
- Domain models live outside Infrastructure and stay business/domain-oriented
- Domain repository contracts are typed on Domain models, not EF scaffolded entities
- EF Core Power Tools generated classes stay in Infrastructure only
- Keep one type per file and use file-scoped namespaces

## Architecture Rules

- Domain owns business/domain models and repository contracts
- Infrastructure owns `DbContext`, scaffolded EF entities, SQL queries, and EF-specific mapping
- Do not expose scaffolded EF entities from Domain contracts
- Translate between scaffolded EF entities and Domain models inside Infrastructure repositories
- Keep shared repository abstractions and batch result types under Domain `Common`

## Operational Workflow

1. **Detect the current state first**
   - Check whether `src\Infrastructures\Client.Project.SqlServer\` already exists.
   - If it does not exist, create the project structure and add the required files.
   - If it already exists, align names, contracts, mappings, configuration, and DI to this standard instead of duplicating artifacts.

2. **Write or update the DDL first**
   - Create or refine a runnable SQL script before touching EF-generated code.
   - Define tables, keys, constraints, indexes, and relationships there first.
   - Keep the DB-first workflow authoritative.

3. **Stop and wait for DB execution**
   - Do not scaffold against an imagined schema.
   - Wait for the developer to execute the script on the target database.

4. **Scaffold with EF Core Power Tools**
   - Use **EF Core Power Tools** after the script has been applied.
   - Let the tool generate entities and the `DbContext`.
   - **Do not generate `DbContext` manually.**
   - Keep generated output in Infrastructure only.

5. **Create or align Domain models**
   - Under `Domain\Models`, add or update business/domain-oriented models that repositories will return and accept.
   - Keep these models free from EF-specific attributes, navigation-loading concerns, and scaffolded naming leakage.

6. **Create or align Domain contracts**
   - Under `Domain\Common\Interfaces`, add shared:
     - `IRepository<TEntity, TKey>`
   - Under `Domain\Common\Results`, add shared batch result types.
   - Under `Domain\Repositories`, create one repository contract per aggregate/model:
     - `ICustomerRepository : IRepository<Customer, int>`
   - `IRepository<TEntity, TKey>` must expose focused CRUD/query methods only.
   - **Do not add `GetAll`.**

7. **Create or align shared SQL infrastructure**
   - Create or refactor `Abstractions\SqlServerBaseRepository.cs`.
   - Centralize shared CRUD helpers, retry behavior, and reusable batch read/write methods there.
   - Keep EF/SQL details in Infrastructure only.

8. **Create or align configuration with Action Pattern**
   - Create or refactor `Configurations\SqlServerConfig.cs` as a **sealed** class.
   - Include:
     - `public const string SectionName = "SqlServer";`
     - connection string setting name/reference
     - `OnReadBatchSize`
     - `OnWriteBatchSize`
     - retry count
   - Put validation in `Validate()`.

9. **Implement concrete repositories**
   - Add or refactor one repository per domain model under `Repositories\`.
   - Inherit from `SqlServerBaseRepository`.
   - Use the scaffolded EF entities internally, then map them to/from Domain models.
   - Implement only model-specific queries/commands on top of shared base behavior.

10. **Register DI with Action Pattern**
    - Add or refactor `ServiceCollectionExtensions.cs`.
    - Expose an Action Pattern entry point such as:
      - `services.AddSqlServerInfrastructure(options => { ... });`
    - In registration:
      1. instantiate `SqlServerConfig`
      2. apply `configure(options)`
      3. call `options.Validate()`
      4. register config, generated `DbContext`, and repositories
    - Do not hardcode connection strings.
    - Use user secrets / secure host configuration; never commit secrets in `appsettings.*`.

## Repository Contract Rules

- One repository contract per domain model or aggregate
- Each contract inherits from `IRepository<TEntity, TKey>`
- `TEntity` must be a Domain model, never a scaffolded EF entity
- No `GetAll`
- Prefer explicit methods such as `GetByIdAsync`, `ExistsAsync`, `CreateAsync`, `UpdateAsync`, `DeleteAsync`, and targeted filtered reads

## Batch Rules

- Batch methods must accept `maxAccumulatedRecords`
- Read/write loops continue until:
  - `maxAccumulatedRecords` is reached, or
  - data is exhausted
- Return a pagination token when the operation is paged
- Do **not** stop the entire loop on per-batch exceptions
- Catch exceptions and accumulate logical failures
- Retry only timeout/connection-type failures, using configured retry count
- Batch results must expose:
  - total operated count
  - success count
  - failed count
  - retry attempts
  - pagination token when relevant

## Constraints

### MUST DO

- Detect whether the SQL Server infrastructure project already exists before generating code
- Start from a runnable DDL script
- Wait for the script to be executed before scaffolding
- Use EF Core Power Tools for entities and `DbContext`
- Keep Domain contracts typed on Domain models, not scaffolded EF entities
- Keep scaffolded EF entities and `DbContext` inside Infrastructure
- Use `SqlServerBaseRepository`, `SqlServerConfig`, and `ServiceCollectionExtensions`
- Use Action Pattern + `Validate()`
- Keep secrets out of source control

### MUST NOT DO

- Do not hand-write the EF `DbContext`
- Do not expose scaffolded EF entities from Domain
- Do not add `GetAll`
- Do not hardcode connection strings
- Do not stop the whole batch loop because one batch fails
